# Current route audit: why LPN samples alone have not decrypted secret.ct

## Decryption equation from code
- `enc_text` stores one wrapped length cipher plus one wrapped cipher per 15-byte plaintext block.
- Each wrapped cipher is `combine(enc(v+m), enc(-m))`.
- For a base layer, public edge aggregation gives a field element proportional to plaintext share times that layer's secret mask `R`.
- Therefore one plaintext block has the form:
  `plain_block = public_T_layer0 / R_layer0 + public_T_layer1 / R_layer1`.

## What `R` needs
Pinned code `pvac_hfhe_cpp@071b0e9` computes:
- `R = PRF_R1 * PRF_R2 * PRF_R3`
- each `PRF_Rx = Toeplitz_x(ybits_x) -> hash_to_fp_nonzero`
- `ybits_x` is generated by LPN: `y = <A,S> xor e`
- both the AES stream for A/e and the Toeplitz stream are keyed by `sk.prf_k` via `derive_aes_key(...)`.

## What upstream published
- 44 files: one per target base layer, domain `pvac.prf.r.1` only.
- They expose A rows and y bits for `PRF_R1` bound to each ciphertext layer.
- They do not expose samples or streams for `PRF_R2`, `PRF_R3`, `TOEP`, `PRF_NOISE*`, or `sk.prf_k`.

## Consequence
- Even if `S` were recovered from public LPN, current public files still do not directly reveal `R_layer` because `R_layer` additionally needs `PRF_R2`, `PRF_R3`, and Toeplitz material keyed by `sk.prf_k`.
- Plaintext-format guessing is blocked because each wrapped block includes an independent random mask `m` and two independent unknown `R` values.
- Public `R_com` in this pinned commit hashes only metadata, not `R`, so it is not an oracle.
- Public `PC` is a Pedersen commitment to `R^{-1}` with hidden randomness derived from `sk.prf_k`, so it is also not a practical verifier.

## Current practical status
No decryption achieved. The remaining promising pivots are: a generator bug in LPN samples, accidental publication of `PRF_R2`/`PRF_R3`/Toeplitz/prf_k material, or a solver disclosure that identifies a non-generic attack.
